# Steering Files

Steering files are the **persistent memory** of your project. They live in `openspec/steering/` and are read silently by `/sdd-apply` before implementing anything.

Their purpose: capture the invisible knowledge that causes PR rejections — naming conventions the linter doesn't catch, layer boundaries not enforced by the build, anti-patterns discovered after painful reviews.

Generated by [`/sdd-init`](../workflow/sdd-init.md) and updated by [`/sdd-steer sync`](../workflow/sdd-steer.md).

---

## The five files

### `product.md`

What the product does, for whom, and the business context.

```markdown
# Product: taskflow

## What it does
A CLI task manager for developers. Manages tasks with priorities,
deadlines, and project groupings from the terminal.

## For whom
Individual developers and small teams who prefer terminal-based workflows.

## Business context
Open source. Users expect Unix philosophy: do one thing well, compose
with other tools via stdin/stdout.
```

Used by `/sdd-apply` to understand intent when making design decisions.

---

### `tech.md`

Tech stack, versions, key libraries, and toolchain.

```markdown
# Tech: taskflow

## Stack
- Language: Python 3.12
- CLI framework: Typer 0.12
- Storage: SQLite via sqlite3 (stdlib)
- Testing: pytest 8.x + pytest-asyncio

## Toolchain
- Dependency management: uv
- Linting: ruff (line-length: 100)
- Build: hatchling

## Key patterns
- Commands are functions decorated with @app.command()
- Storage layer isolated in src/taskflow/db/
```

---

### `structure.md`

Codebase layout — which directories contain what.

```markdown
# Structure: taskflow

src/taskflow/
├── cli/          ← Typer commands (entry points)
├── core/         ← Business logic (no I/O)
├── db/           ← SQLite storage layer
└── models.py     ← Dataclasses

tests/
├── unit/         ← Pure logic tests
└── integration/  ← CLI tests via runner
```

---

### `conventions.md`

Naming rules, patterns, and anti-patterns specific to this project.

```markdown
# Conventions: taskflow

## Naming
- Commands: verb_noun (add_task, list_tasks, delete_task)
- Models: singular PascalCase (Task, Project, Tag)
- DB functions: snake_case with table prefix (task_create, task_get_by_id)

## Patterns
- All CLI commands call a core function — no business logic in CLI layer
- DB functions return dataclasses, never raw tuples
- Errors are raised as TaskflowError subclasses, caught in CLI layer

## Anti-patterns
- No direct sqlite3 calls outside db/ directory
- No print() in core/ — only in CLI layer
- No global state — pass dependencies explicitly
```

This is the most important steering file. It prevents the most common review rejections.

---

### `project-rules.md`

Explicit rules stated as imperatives for `/sdd-apply` to follow.

```markdown
# Project Rules: taskflow

1. Every new command MUST have a corresponding unit test
2. DB schema changes MUST include a migration script in db/migrations/
3. New models MUST be added to models.py, not inline in other files
4. CLI help text MUST start with a verb (Add, List, Delete, Show)
5. Error messages MUST include the invalid value and the expected format
```

These complement `conventions.md` for cases where a rule is absolute and non-negotiable.

---

## Keeping steering current

Steering files drift over time. Run `/sdd-steer sync` periodically to detect and update outdated rules.

Common triggers for an update:
- A new library is added to the stack
- A convention is established after a sprint retro
- A new anti-pattern is discovered in review

Stale steering is worse than no steering — it generates confidently wrong code.
